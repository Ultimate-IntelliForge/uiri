# Implementation Plan: UIRI CMS Platform

**Branch**: `001-cms-platform` | **Date**: 2026-05-22 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/001-cms-platform/spec.md`

---

## Summary

Build the ULTIMATE INTELLIFORGE RESEARCH INSTITUTE (UIRI) CMS platform — a Turborepo
monorepo consisting of a NestJS REST API (`apps/api`) and a Next.js public
site + admin dashboard (`apps/web`). The platform serves as the institute's public face
and allows authorised staff to manage all content (departments, publications, team
profiles, pages, media) without any code deployment. Content flows through a
Blog Manager → Department Admin approval workflow before going live. Private content is
gated by RBAC and one-use encrypted links for external access.

The implementation is divided into **12 batches**. Each batch has its own git branch,
is independently testable, and is merged into `main` only after user approval.

---

## Technical Context

**Language/Version**: TypeScript 5.x (`strict: true`) — both apps and all packages.

**Primary Dependencies**:
- `apps/api`: NestJS 10, Prisma 5, Passport.js, passport-local, passport-jwt,
  @nestjs/jwt, @nestjs/throttler, class-validator, class-transformer, googleapis
- `apps/web`: Next.js 15 (App Router), next-intl, react-hook-form, zod, shadcn/ui,
  Tailwind CSS v4, Lucide React, Framer Motion
- `packages/types`: zod (shared schema definitions)
- `packages/ui`: shadcn/ui components, Radix UI, cn utility
- `packages/config`: ESLint config, TypeScript base configs, Tailwind base config

**Storage**: Supabase (PostgreSQL free tier) via Prisma — `DATABASE_URL` (pooler) +
`DIRECT_URL` (direct, for migrations only). Google Drive API (Service Account) for
file/document storage in v1.

**Testing**: Jest (unit — NestJS services/guards), Playwright (e2e — critical public
flows). Tests are **not** mandatory for v1 given startup scale; each batch notes
what to manually verify.

**Target Platform**: Vercel (`apps/web`), Railway/Render (`apps/api`).

**Performance Goals**:
- Public pages: LCP < 2.5 s, CLS < 0.1, INP < 200 ms (Core Web Vitals).
- API response time: < 200 ms p95 for read endpoints under current load.

**Constraints**:
- All secrets must stay server-side (never in browser bundle).
- `SameSite=Strict` cookies — auth tokens never sent cross-origin.
- No magic schema migrations — always `prisma migrate dev`.

**Scale/Scope**: 5 admins, ~50 monthly public visitors at launch; architecture must
not prevent horizontal scaling.

## Batch Workflow

Every batch follows this lifecycle:

```
1. Cut branch from main:   git checkout -b ###-batch-N-description main
2. Implement all tasks in the batch
3. Manually verify the batch acceptance checklist
4. Open for review → USER APPROVES
5. Merge to main:          git checkout main && git merge ###-batch-N-description
6. Cut next batch from updated main
```

No batch is merged without explicit user approval.
